# 项目可改进点与解决方案

## 1. 回收站清理链路可靠性
- 现状问题：目前已引入 RabbitMQ 异步清理，但消费者失败重试、死信队列、幂等键仍可继续增强。
- 风险影响：极端情况下可能出现重复消息、长时间重试堆积。
- 解决方案：
  - 为 `recycle.expired.queue` 增加 DLX 与重试队列（如 3 次后进入死信）。
  - 消费逻辑加入幂等记录（例如 `cleanup_log(file_id, status)`）。
  - 增加告警：队列积压数量、死信数量、消费延迟。

## 2. 后端分层与事务边界
- 现状问题：当前 Service 层仍有部分查询拼装和流程编排耦合在一起。
- 风险影响：模块变更时影响面较大，单元测试成本高。
- 解决方案：
  - 按“应用服务（编排）/领域服务（规则）/仓储（数据）”继续拆分。
  - 所有跨表写操作统一由应用服务管理事务。
  - 新增 `internal/app` 进行依赖装配，Router 仅负责路由注册。

## 3. SQL 与索引优化
- 现状问题：`favorite`、`share`、`file` 部分高频查询未统一覆盖组合索引。
- 风险影响：数据量上来后列表/搜索变慢。
- 解决方案：
  - 为高频条件建立组合索引，例如：
    - `file(user_id, is_deleted, parent_id, updated_at)`
    - `favorite(user_id, created_at)`
    - `share(user_id, is_deleted, created_at)`
  - 接口层加分页上限，避免大页查询。

## 4. API 一致性与错误码
- 现状问题：不同接口返回结构与错误语义有差异（如 message 文案、状态码映射）。
- 风险影响：前端需要写大量分支兼容。
- 解决方案：
  - 统一 `code/message/data/requestId` 响应体。
  - 建立错误码字典（鉴权、参数、业务、存储、系统）。
  - 网关/中间件注入 `requestId`，日志链路统一追踪。

## 5. 前端复用与状态管理
- 现状问题：页面头部、文件图标映射、空态视图存在重复实现。
- 风险影响：需求变更时需要多处同步，容易漏改。
- 解决方案：
  - 已抽取 `PageHeader` 与 `fileIcon` 工具，建议继续抽取：
    - 通用 `FileTable`（操作列通过插槽扩展）
    - 通用 `EmptyState` 组件
    - 文件操作 `useFileActions` 组合函数
  - API 层增加 DTO 适配器，页面直接消费统一字段。

## 6. 测试与质量门禁
- 现状问题：核心流程缺少端到端回归（上传/删除/恢复/分享）。
- 风险影响：重构后回归风险高。
- 解决方案：
  - 后端增加 service/repository 单测（含事务回滚场景）。
  - 增加最小 e2e（登录、上传、回收站、清理）。
  - CI 中加入：`go test`、前端 lint、构建检查、迁移校验。

## 7. 可观测性
- 现状问题：日志以文本为主，缺少结构化字段和关键指标。
- 风险影响：线上定位慢。
- 解决方案：
  - 使用结构化日志（requestId/userId/fileId/event）。
  - 暴露指标：上传耗时、MinIO失败率、MQ积压、清理吞吐。
  - 增加慢查询日志与阈值告警。

## 8. 配置与安全
- 现状问题：示例配置含明文密码与内网地址，缺少密钥管理规范。
- 风险影响：泄露风险与环境迁移成本。
- 解决方案：
  - 密钥改用环境变量/密钥管理服务，配置文件仅保留占位符。
  - 增加配置校验（启动时检查必填项、非法值）。
  - 生产环境禁用调试级日志，开启最小权限访问策略。
